// WORKFLOW STEPS

Click each step to visualize the data flow in the Integration-Manager model

1
Maintainer Pushes to Public
The project maintainer pushes their code to the blessed (canonical) public repository.
MAINTAINER
2
Contributor Clones Repository
A contributor clones the blessed repository to their local machine and makes changes.
CONTRIBUTOR
3
Contributor Pushes to Fork
The contributor pushes their changes to their own public copy (fork) of the repository.
CONTRIBUTOR
4
Pull Request Notification
The contributor sends the maintainer a notification (email/PR) asking them to pull the changes.
CONTRIBUTOR MAINTAINER
5
Maintainer Pulls & Merges
The maintainer adds the contributor's repo as a remote, pulls changes, tests locally, and merges them.
MAINTAINER
6
Push to Blessed Repository
The maintainer pushes the merged changes back to the blessed (canonical) repository.
MAINTAINER
// KEY BENEFITS
  • Common on GitHub, GitLab, and similar platforms
  • Easy to fork and push changes publicly
  • Developers work independently at their own pace
  • Contributors don't wait for integration
  • Maintainer controls what enters the canonical repo
