Workflow Steps
1 Maintainer Pushes to Public
The project maintainer pushes to their public repository, which serves as the main reference point for the project.
2 Contributors Clone & Fork
Contributors clone the maintainer's repository and create their own public forks where they have write access.
3 Contributors Make Changes Locally
Contributors work on their local clones, making changes and commits at their own pace.
4 Push to Public Forks
Contributors push their changes to their own public forks, making them visible to everyone.
5 Request Pull (Notify Maintainer)
Contributors send a pull request or email to the maintainer, asking them to review and pull their changes.
6 Maintainer Reviews & Merges
The maintainer adds the contributor's repository as a remote, tests changes locally, and merges them.
7 Push to Canonical Repository
The maintainer pushes merged changes to the canonical (blessed) repository, making them official.
Key Benefits
  • Each developer has write access to their own public repository
  • Everyone has read access to all other repositories
  • Contributors work at their own pace without blocking
  • Easy to fork and share changes publicly
  • Maintainer controls what enters the official project
  • Very common in GitHub and GitLab workflows
